# Кимби СУ фреймворк / kimbi-ms-framework — фреймворк системы управления

Версия: 0.1.0

Абстрактный фреймворк для создания системы управления без ограничений
по технической реализации

## Назначение

Кимби СУ — фреймворк, помогающий создать систему управления. Идея в том, что
основой управления является единая информационная система. В ней объединяются
планы, знания и факты, что даёт целостность, достаточную для управления.
Преимущество такого подхода раскрывается при использовании ИИ, потому что ему
становятся доступны все сферы предмета управления через информацию.

Во главе угла стоят **цели**. Всё остальное существует для успешной постановки
и достижения целей.

## Сущности

### Цель

Желаемое состояние мира с измеримым критерием достижения.

Структура цели:
- **название** — формулировка желаемого состояния (свершившийся факт);
- **описание** — контекст: почему эта цель, что вокруг неё;
- **критерии успеха** — по чему проверяем, что состояние достигнуто;
- **желаемая дата достижения** — когда хотелось бы достичь; берётся из желания
(своего или заказчика) и обоснования не требует;
- **плановая дата достижения** — когда реально планируется достичь; ставится
не из желания, а из предпосылок — у плановой даты должно быть обоснование;
- **дата достижения** — фактическая;
- **статус** — текущий статус. варианты:
  - **формулируется** — цель формулируется;
  - **сформулирована** — формулировка готова, цель ещё не поставлена;
  - **поставлена** — цель поставлена;
  - **завершена** — работа над целью завершена. У завершённой цели
    обязательны исход и результат;
- **отложена** — признак: достижение цели приостановлено. Не отдельный статус —
статус при откладывании сохраняется, поэтому видно, на каком этапе цель
отложена, а при возврате из отложенных статус не нужно выбирать заново;
- **заметки** - любая не структурированная информация, которая появляется в ходе работы над целью
и может быть полезна для её достижения
- **журнал** - фиксация действий с временной меткой в ходе работы над целью
- **исход** — с каким исходом завершена. обязательно к заполнению в финальном статусе. варианты:
  - **успех** — цель достигнута;
  - **провал** — не достигнута;
  - **отмена** — потеряла смысл, изменились приоритеты;
- **результат** — фактический результат: что получилось, почему не получилось
или почему отменили. обязательно к заполнению в финальном статусе при любом исходе;
- **связи** — у цели бывает связь **способствует** → одна или несколько целей
(см. раздел «Связи»).

Достижение цели:
Для достижения цели существуют разные средства, например, задачи. Однако,
успешное выполнение всех действий для достижения цели не означает достижение
цели. Достижение цели проверяется только по критериям успеха.

Декомпозиция целей:
Иерархии целей нет — цели плоские. Крупная цель декомпозируется через связь
«способствует»: каждая способствующая цель — полноценная цель со своими
критериями успеха и своим жизненным циклом, а не пункт чек-листа. Если
формулируемая «подцель» не выдерживает этого требования — значит, это задача.

### Задача

Задача может быть средством достижения цели, быть подзадачей или просто
самостоятельной задачей, не связанной ни с чем.

Структура задачи:
- **название** — формулировка того, что нужно сделать;
- **описание** — контекст: зачем, что важно учесть;
- **желаемая дата начала** — когда хотелось бы начать; берётся из желания
(своего или заказчика) и обоснования не требует;
- **плановая дата начала** — когда реально планируется начать; ставится
не из желания, а из предпосылок — у плановой даты должно быть обоснование;
- **дата начала** — фактическая;
- **желаемая дата завершения** — когда хотелось бы завершить; берётся из желания
(своего или заказчика) и обоснования не требует;
- **плановая дата завершения** — когда реально планируется завершить; ставится
не из желания, а из предпосылок — у плановой даты должно быть обоснование;
- **дата завершения** — фактическая;
- **статус** — текущий статус. варианты:
  - **планируется** — задача формулируется и планируется, выполнение не начато;
  - **на старте** — подготовлена, ожидает даты начала;
  - **в работе** — задача выполняется;
  - **завершена** — работа над задачей завершена. У завершённой задачи
    обязательны исход и результат;
- **приостановлена** — признак: выполнение задачи приостановлено. Не отдельный
статус — статус при приостановке сохраняется, поэтому видно, на каком этапе
задача приостановлена, а при возобновлении статус не нужно выбирать заново;
- **заметки** - любая не структурированная информация, которая появляется в ходе работы над задачей
и может быть полезна для её выполнения
- **журнал выполнения** - фиксация действий с временной меткой в ходе выполнения задачи
- **исход** — с каким исходом завершена. обязательно к заполнению в финальном статусе. варианты:
  - **успех**
  - **провал**
  - **отмена**
- **результат** — фактический результат. обязательно к заполнению в финальном статусе при любом исходе;
- **связи** — у задачи бывают связи **способствует** → одна или несколько целей
и одна или несколько задач, **подзадача** → задача, **зависит от** → задача
(см. раздел «Связи»).

### Проект, процесс

Зарезервированы на будущее.

### Знание

Описание мира: продукта, метода, домена, рынка... Знание само по себе ничем
не управляет, но может быть использовано любыми сущностями по своему усмотрению.

- **база знаний** - пространство, в котором хранятся знания
- **ценность** - база знаний — не свалка черновиков, а описание мира, имеющее пользу
- **dry (don't repeat yourself)** - смысл фиксируется один раз в одном документе, 
остальные ссылаются, а не дублируют
- **принцип создания пространств** - знания могут преобразовываться во что-то бОльшее,
чем просто знания, получая функциональность. Например, "Маркетинг" может получить свою
стратегию и свои инструменты (сайт, бот...). В таком случае, создаётся новое пространство
в системе управления, знания переносятся туда, а в базе на их месте остаётся информация:
что это, зачем оно, где живёт. Информация не дублируется, база сохраняет знание —
через ссылку и суть
- **единая точка входа.** Ни одно пространство не существует,
если о нём нельзя узнать из базы знаний.
- **векторизация** - база знаний должна быть пригодна для векторизации;
выжимки-указатели и отсутствие дублей — то, что обеспечивает эту пригодность
без специальной подготовки.

## Связи

Смысл каждого типа связи фиксируется здесь один раз; в описаниях сущностей —
только указатели на типы, без повторения семантики.

- **способствует** — «вносит вклад в достижение/выполнение». Направлена от
способствующей сущности к той, которой она помогает: цель → цель,
задача → цель, задача → задача. Many-to-many: одна сущность может
способствовать нескольким и наоборот. Поведения не несёт: завершение
способствующих не завершает цель — достижение цели проверяется только
по её критериям успеха.
- **подзадача** — «является частью работы». задача → задача, у подзадачи
один родитель. Завершение всех подзадач не завершает родительскую задачу
автоматически: исход и результат заполняет человек.
- **зависит от** — «нельзя выполнить, пока не готово то». задача → задача.
Циклы запрещены. Незакрытая зависимость при старте задачи — повод для
предупреждения, а не запрет.

## Развитие и расширение

Главный принцип — **называть вещи своими именами и не ограничиваться**.
Модель фреймворка жизнеспособная, но если покажется, что что-то стоит
назвать по-другому, расширить или изменить — фреймворком это только приветствуется.
Ценность фреймворка в том, что он целостный и полезный, и с него можно смело начинать.

Система управления может расширяться вместе с развитием предмета управления.
Принципиально для этого существует несколько способов и выбор зависит от ситуации:
- **расширение самой системы управления** — например, добавление пространства
Маркетинг
- **интеграция с другими системами**
-  **комбинация обоих подходов**
