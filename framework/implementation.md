# Реализация Кимби СУ фреймворк на markdown и git

Как фреймворк Кимби СУ реализуется в виде директорий с md-файлами и git.
Понятия и модели — [framework.md](framework.md) (текст реализуемой версии
фреймворка, лежит рядом); здесь — принципы носителя: файлы, директории,
форматы, процедуры, git.

Канонически фреймворк развивается в репозитории
[kimbi-ms-framework](https://github.com/cheburatino/kimbi-ms-framework);
локальная копия не редактируется на месте, а обновляется оттуда —
см. [README](../README.md).

## Оформление документов

- О документе должно быть понятно из естественной информации о нём:
  расположение, имя файла, заголовок, содержание. Специальные вводные
  блоки (цитаты `>` с описанием сути) не используются.
- Если назначение документа требует пояснения — это делается обычным
  текстом или отдельным заголовком в теле документа.
- Принадлежность к пространству выражается расположением (документ лежит
  в директории пространства), связь с корневым документом — ссылкой из
  корневого документа, а не припиской в каждом аспекте.

Так документы гармоничнее и проще мигрируют из md в другие реализации.

## Принципы носителя

- **Вся информация — в markdown-файлах.** Текст — это данные: диффы, поиск,
  правки человеком и агентом одними и теми же инструментами.
- **Структурные поля объекта — в YAML front matter**, свободный текст
  (описание, критерии, заметки, журнал, результат) — в теле файла.
  Даты — `YYYY-MM-DD`.
- **Каждое пространство — директория** в корне системы.
- **Сортировка — числовым префиксом без точки** в имени директории
  (`1 ...`, `2 ...`, `3 ...`): порядок виден в любом файловом менеджере
  и в выводе `ls`.
- **Пустые директории — часть структуры.** Git пустые директории не хранит,
  поэтому в пустой директории лежит файл `.gitkeep`.
- **Точка входа — `README.md` в корне системы:** что это за директория
  и ссылки на описание системы.

## Цели и задачи

- Каждая цель и задача — **отдельная директория** со слагом (латиница,
  kebab-case). Внутри — файл объекта (`goal.md` / `task.md`) и любые рабочие
  файлы: материалы для решения, черновики, файлы с результатами. Всегда
  директория, даже если внутри пока один файл — путь не меняется при
  появлении рабочих файлов.
- **Слаг — идентификатор, а не название.** Он создаётся по мотивам названия
  один раз, при рождении объекта, и дальше за названием не следует;
  человекочитаемое название живёт в `title` front matter и меняется
  свободно. Расхождение слага с названием допустимо: слаг читается как
  адрес, имя смотрят в `title`. Смена названия — правка одной строки;
  смена слага ломала бы все входящие ссылки. Если слаг стал вводить
  в заблуждение — по смыслу это уже другой объект: старый завершается
  с исходом `canceled`, вместо него создаётся новый.
- **Статус кодируется статусной директорией** с числовым префиксом —
  состояние дел видно сразу, без открытия файлов. Наборы статусов у цели
  и задачи разные (см. framework.md); в носителе статусы кодируются
  латинскими слагами:

  | Цель | директория | Задача | директория |
  |---|---|---|---|
  | формулируется | `1 formulating` | планируется | `1 planning` |
  | сформулирована | `2 formulated` | на старте | `2 ready` |
  | поставлена | `3 set` | в работе | `3 in_progress` |
  | завершена | `4 ended` | завершена | `4 ended` |

  Исходы кодируются так же: успех — `success`, провал — `failed`,
  отмена — `canceled`. Front matter дублирует статус; источник истины
  при расхождении — front matter.
- **Признак «отложена» (цель) / «приостановлена» (задача) — поле
  `suspended`** во front matter: дата установки признака, пустое поле —
  объект активен. Статусная директория при установке признака не
  меняется — статус сохраняется, виден этап, на котором объект
  приостановлен; снятие признака — очистка поля.

Front matter цели (`goal.md`):

```yaml
---
title: Название цели
status: formulating      # formulating | formulated | set | ended
outcome:                 # success | failed | canceled — только при ended
suspended:               # дата откладывания; пустое поле — цель активна
desired_achieve_date:
plan_achieve_date:
end_date:                # дата завершения — при любом исходе
contributes_to:          # список путей к целям, которым способствует
---
```

Front matter задачи (`task.md`):

```yaml
---
title: Название задачи
status: planning         # planning | ready | in_progress | ended
outcome:                 # success | failed | canceled — только при ended
suspended:               # дата приостановки; пустое поле — задача активна
desired_start_date:
plan_start_date:
start_date:
desired_end_date:
plan_end_date:
end_date:
contributes_to:          # список путей к целям и задачам
parent_task:             # путь к родительской задаче — только у подзадачи
depends_on:              # список путей к задачам, от которых зависит
---
```

Связи фреймворка кодируются полями front matter:

- **способствует** — `contributes_to`: YAML-список путей (у цели —
  к целям, у задачи — к целям и задачам). Поле необязательно:
  самостоятельная задача без связей легальна;
- **подзадача** — `parent_task`: одиночный путь к родительской задаче;
- **зависит от** — `depends_on`: YAML-список путей к задачам. Циклы
  запрещены; незакрытая зависимость при старте задачи — предупреждение,
  а не запрет.

Знания, которыми руководствуется цель или задача, привязываются ссылками
в тексте (описание, критерии, результат) — там, где виден смысл связи;
отдельного структурного поля для этого нет.

Новые цели и задачи создаются копированием из шаблонов:
[../templates/goal.md](../templates/goal.md),
[../templates/task.md](../templates/task.md).

## Ссылки

- Пути в ссылках и полях front matter указываются **от корня системы** —
  так исходящие ссылки объекта не зависят от его собственного положения.
- Статус входит в путь (статусная директория), поэтому смена статуса —
  атомарная процедура:
  1. обновить front matter (`status`, при завершении — `outcome`, даты,
     заполнить результат);
  2. переместить директорию объекта в соответствующую статусную директорию;
  3. найти по слагу все входящие ссылки и обновить в них путь.

  Механическую часть процедуры выполняет скрипт
  [../scripts/change_status.py](../scripts/change_status.py) —
  см. «Скрипты».

## Скрипты

Механические процедуры автоматизированы скриптами в
[../scripts/](../scripts/). Требование — python3 (наряду с git это
системные требования реализации). Смысловые шаги — формулировки,
заполнение результата, коммит — остаются за человеком или агентом.

- `init_instance.py <путь>` — создание инстанса: структура директорий,
  README, точки входа для ИИ-агентов, указатель на реализацию,
  git-репозиторий с первым коммитом `init`;
- `change_status.py <путь-к-объекту> <статус> [--outcome …]` — смена
  статуса: front matter (`status`, `outcome`, фактические даты),
  перемещение директории, обновление входящих ссылок по всей системе.

## Git

- Система живёт в **собственном git-репозитории**: у управления своя
  история, не смешанная с историей других проектов.
- **История — журнал и аудит** системы: кто, когда и что менял.
  История не переписывается.
- Работа ведётся **в одной ветке**: состояние системы всегда одно,
  ветвление не используется.
- **Коммиты — по смысловым операциям** (создана цель, завершена задача,
  обновлено знание), а не по файлам. Сообщение коммита отвечает на вопрос
  «что произошло в управлении», а не «какие файлы изменились».
- **Удалённый репозиторий — резервная копия** и точка синхронизации.
