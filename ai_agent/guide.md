# Руководство для ИИ-агента

Как агенту работать в системе управления на md-реализации Кимби СУ.
Документ вендоронезависим: подходит любому агенту. Вендорские точки входа
(CLAUDE.md, AGENTS.md и т.п.) — однострочные указатели на README инстанса.

## Начальная загрузка

Прочитать в этом порядке:

1. `README.md` инстанса — что это за инстанс;
2. `framework/framework.md` в клоне kimbi-ms-md — понятия, статусы,
   связи (текст реализуемой версии фреймворка);
3. `framework/implementation.md` там же — устройство системы на md и git.

Где лежит локальный клон kimbi-ms-md, говорит указатель
`1 knowledge/tech/kimbi-ms.md` в базе знаний инстанса; соглашение по
умолчанию — рядом с директорией инстанса.

Дальше контекст собирается под сценарий работы, а не чтением всего подряд.

## Правило сборки контекста: один шаг по ссылкам

Стартовый контекст — **один шаг по ссылкам от рабочих объектов**; глубже —
**по требованию**, а не по цепочке.

Ссылки внутри прочитанных знаний — не команда «читай дальше», а карта:
такое знание существует и где лежит. Дочитывать их надо в момент, когда
конкретное действие этого требует.

Почему одного шага достаточно: смысл фиксируется один раз (прямая ссылка
ведёт к самому смыслу, а не к цепочке пересказов), указатели несут выжимку
(по тексту вокруг ссылки понятно, нужно ли туда идти). Если для понимания
документа регулярно приходится открывать его ссылки — это сигнал чинить
документ (смысл размазан или продублирован), а не увеличивать глубину
обхода.

## Сценарий: работа над целью

Стартовый контекст:

1. **Цель**: `goal.md` и рабочие файлы в директории цели (журнал прогресса
   и т.п.);
2. **Задачи цели**: поиск по полю `goal:` со слагом цели; у найденных —
   подзадачи по полю `parent_task:`. Целиком читаются задачи в статусах
   `draft` и `in_progress`; завершённые — списком заголовков, их файлы —
   по требованию;
3. **Знания по прямым ссылкам** из текста цели и активных задач
   (описание, критерии и т.п.).

На этом загрузка останавливается — дальше по правилу одного шага.

Обратные связи (какие цели/задачи ссылаются на данное знание) в стартовый
контекст не входят; при необходимости ищутся поиском по пути документа.

## Сценарий: работа над задачей

Стартовый контекст:

1. **Задача**: `task.md` и рабочие файлы в директории задачи;
2. **Контекст «зачем»**: цель по полю `goal:`; у подзадачи — цепочка
   родительских задач по `parent_task:` вверх до цели. Читаются файлы
   объектов цепочки (`task.md`, `goal.md`), их рабочие файлы —
   по требованию;
3. **Подзадачи задачи**: поиск по полю `parent_task:` со слагом задачи.
   Целиком читаются подзадачи в статусах `draft` и `in_progress`;
   завершённые — списком заголовков;
4. **Знания по прямым ссылкам** из текста задачи.

Соседние задачи той же цели в стартовый контекст не входят — они уже
учтены при работе над целью, где задача ставилась.

Смена статуса и фиксация результата — по процедуре из
`framework/implementation.md`. Выполнение задачи не равно достижению цели:
при завершении фиксируется результат задачи, а вывод о цели делается
в сценарии работы над целью.

## Сценарий: работа над базой знаний

Стартовый контекст:

1. **Рабочий документ** знания (или документы, если правка затрагивает
   несколько);
2. **Прямые ссылки** рабочего документа — как карта, по правилу одного
   шага;
3. **Входящие ссылки** — поиском по пути документа: кто опирается на это
   знание. Нужны при изменении документа, при чтении — нет.

Правки подчиняются принципам раздела «База знаний» во `framework.md`;
главный — смысл фиксируется один раз, дубли сводятся к источнику.
Переименование или перемещение документа включает обновление всех
входящих ссылок — в той же операции, а не «потом».

## Сценарий: работа над самой системой управления

Стартовый контекст — документы системы целиком:

1. `framework/framework.md` — понятия и модели (текст реализуемой версии
   фреймворка);
2. `framework/implementation.md` — носитель: файлы, форматы, процедуры;
   при изменении форматов — шаблоны в `templates/` и скрипты в `scripts/`;
3. `ai_agent/guide.md` — этот документ.

Изменение фиксируется на своём уровне: новое понятие или связь — во
фреймворке; формат, процедура, устройство носителя, скрипты — в
реализации; правило работы агента — в этом руководстве. Локальные
договорённости конкретного инстанса — в README инстанса, не в документах
системы.

Канонически фреймворк развивается в репозитории kimbi-ms-framework.
Копия `framework/framework.md` не редактируется на месте: концептуальное
изменение вносится в канон, затем одной операцией копия обновляется
и реализация приводится в соответствие. Осознанная кастомизация
фреймворка под себя — форк kimbi-ms-md: там копия и реализация меняются
вместе.

Изменение системы затрагивает инстансы: при обновлении локального клона
существующие объекты инстанса (цели, задачи, знания) приводятся к новому
устройству сразу, одной операцией, а не остаются в старом формате.
